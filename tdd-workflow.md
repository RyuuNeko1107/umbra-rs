# TDDワークフロー

`$ARGUMENTS` を、以下の役割分離フローで実装する。
各工程は前工程を無条件に追認せず、独立した視点で検証する。

## モード（工程0で判定し、理由を明記する）

| モード | 工程 |
| --- | --- |
| **trivial**（typo / コメント / 設定値 / 機械的リネーム） | 既存テスト確認 → 実装 → 関連テスト＋静的解析 |
| **standard**（通常の小〜中規模実装・バグ修正） | 工程0 →（1+2: テスト設計兼作成）→ 3 → 4 → 5 → 6。工程7は軽い通過確認のみ |
| **strict**（後方互換を壊しうる公開仕様変更・永続データ形式変更・認証/認可/機密情報/課金/破壊的操作・重大障害につながる並行処理など、**高影響または復旧困難**な変更） | standardの各工程を独立分離 ＋ カバレッジ計測・ミューテーション（工程7完全版）・並列/順序変更/複数回実行・セキュリティ/依存検査を必須化 |

迷ったら standard。普段使いで毎回フル装備にしない。

## 基本ルール

- **実験・検証はDocker内で実行**: red/green確認、全テスト、静的解析、カバレッジ、ミューテーション、試行的な検証コマンドは原則としてDocker/Compose内で実行する。ローカル直接実行は、Docker自体の疎通確認・エディタ整形・Dockerで再現不能な環境問題の切り分けなどに限定し、最終報告に理由を残す。
- **追認禁止**: レビュー担当には、確定仕様・レビュー対象成果物・独立検証に必要なコードとテストを渡す。前工程の自己評価・判断理由・期待する結論は渡さない。隠すのは情報ではなく前任者の結論。
- **サブエージェント分離**: テスト設計/作成・各レビューは Task tool でサブエージェントに委譲する。渡す文脈は以下に限定する。
  - テスト担当には **確定仕様・公開インターフェース・関連既存テスト**、および **ハーネスや既存挙動の理解に必要な範囲の既存コード** を渡す。**実装案・予定差分・メインの設計判断は渡さない**。
  - レビュー担当には **確定仕様・レビュー対象成果物・検証に必要な関連コード** を渡す。**作成者の自己評価・判断理由・期待する結論は渡さない**。
  - 隠すのはコード情報ではなく、前任者の結論と正当化。コードまで隠すとレビュー精度が落ちる。
  - 同一セッションで兼任する場合も、観点を明示的に切り替え、前工程の結論を前提にしない。
  - **サブエージェント機能を使えない環境**では、役割ごとに独立したレビュー入力を再構成し（前工程の自己評価・結論を除外）、別レビューとして実施する。完了とはせず「物理的な担当分離なし」を残存リスクとして最終報告に記載する。
  - **trivial ではサブエージェントによるテスト設計・レビューを必須としない**。ただし変更が想定より広い／挙動変更を含む／テスト追加が必要と判明したら standard へ昇格する。
- **黙って直さない**: テストに問題があると判断したら、実装側で書き換えず、問題点を明示して該当工程に戻す。
- **ループ上限**: 1周＝ある工程から前工程へ差し戻し、修正後に元の工程へ再到達するまで。同じレビュー工程で3回目の差し戻しが必要になった時点で、問題点と選択肢を整理して人間に判断を仰ぐ。加えて、**1タスク累計4回（工程横断）**差し戻したら、個別工程の上限未満でも一度立ち止まって人間に相談する（数値はプロジェクトで調整可）。
- **コミット**: ユーザーからコミットを許可されている場合のみ red と green を別コミットに分ける。許可がない場合はコミットせず、実行ログと差分で証跡を残す。

役割の割り当て:

| 工程 | 担当 |
| --- | --- |
| テスト設計・作成 | テスト担当サブエージェント |
| テストレビュー | レビュー担当サブエージェント（テスト作成者とは別） |
| 実装 | メインエージェント |
| 実装レビュー | レビュー担当サブエージェント（実装者とは別） |
| テスト結果レビュー | テスト作成者・実装者とは別（strict: 別サブエージェント / standard: 別観点可） |

---

## 工程0: 仕様確定とモード判定（メインエージェント）

1. 仕様・既存コード・既存テストを読む。
2. 曖昧点を洗い出す。**実装結果・公開仕様・データ互換性・安全性に影響する曖昧点のみ人間に確認する**。軽微な不明点は、仮定を明示して進める（仮定は報告に残す）。
3. モードを判定する。**裁量ではなく機械的トリガーで決める**: モード表の strict 条件のいずれかに該当したら自動的に strict とする。トリガーが立った strict を降格できるのは、実装者の裁量ではなく**人間の承認がある場合のみ**。standard と判定したときは「なぜ strict でないか」を**最終報告に必須記載**する（被検証者がランクを選ぶ構造を、後から人間が監査できるようにするため）。

この段階では実装コードを変更しない。

## 工程1: テスト設計（テスト担当サブエージェント / standardでは工程2と統合可）

仕様・既存コード・既存テストを確認し、外部から観測できる振る舞いを基準にテスト計画を作る。内部構造には依存しない。

> standard: テスト担当がテスト計画を簡潔に提示した後、同一サブエージェントが続けてテストを作成してよい。計画はテストコードとともに工程3のレビュー対象とする（頭の中だけで済ませない）。

含めるもの:
- 正常系 / 境界値 / 異常系
- 状態遷移（未到達経路を含む）
- 既存機能への回帰
- 並行・非同期がある場合: 競合・順序・再実行
- 永続化・外部連携がある場合: 失敗・復旧
- 必要に応じて性能・セキュリティ（不要と判断したら理由を明記）

実装コードは変更しない。

## 工程2: テスト作成（テスト担当サブエージェント）

レビュー前のテストコードを書く。条件:
- 実装詳細に可能な限り依存しない
- 仕様上重要な出力・状態・副作用を明確に検証する
- テスト間で共有状態・実行順序に依存しない
- 時刻・乱数・外部サービスなど不安定要因を制御する
- モックは最小限。呼び出し回数だけでなく最終的な振る舞いも検証する
- 特定の実装方法を強制しない

実装前に実行し、新規要件のテストが**想定どおりの理由で失敗する**ことを確認する（red）。
- **red確認のために既存実装を一時改変・無効化してはならない**。現在の実装に対して新規テストを実行し、未実装要件が期待した理由で失敗することを確認する。
- 既存実装で既に満たされる場合は、その理由を記録し green 開始時点として扱う。red が成立しないなら red コミットは作らず、理由を残す。

## 工程3: テストレビュー（テスト作成者とは別のサブエージェント）

テスト設計とコードを、作成者の判断を前提にせず検証する。重点:
- 仕様の抜け・解釈違い
- 境界値の不足
- 異常系・失敗時挙動の欠落
- 未検証の状態遷移経路
- 弱すぎるアサーション / 過剰なモック
- 実装を追認するだけのテスト
- 実行順序・共有状態への依存
- 偶然成功しやすい条件
- 成功ケースしか見ていない / エラーの種類・内容を検証していない
- 回帰テストの不足
- テスト自体の誤り・無意味な検証

不足があればテスト担当に修正させ、再レビュー（差し戻しはループ上限に従う）。

## 工程4: 実装（メインエージェント）

レビュー済みテストと仕様全体を満たす実装を行う。
- テストを通すためだけのハードコード・特定入力専用分岐を入れない
- 既存の設計・命名・責務分離・エラー処理方針に合わせる
- 変更範囲を必要以上に広げない
- サブエージェントのテストを理由なく書き換えない。問題があれば工程3へ戻す
- 公開挙動・設定・操作方法が変わる場合は、関連ドキュメント（README・API仕様・変更履歴・マイグレーション）も更新する

## 工程5: 実装レビュー（実装者とは別のサブエージェント）

差分と関連コードを検証。重点:
- 仕様逸脱 / 未処理の分岐・入力
- エラー処理 / 状態遷移の矛盾
- 既存機能への副作用 / 後方互換性
- 公開挙動変更時のドキュメント追従漏れ（README・API仕様・変更履歴・マイグレーション）
- 性能劣化 / 不要なI/O・重複処理
- 並行処理上の問題
- セキュリティ（入力検証・権限確認・機密情報漏えい）
- 可読性・責務分離・拡張性・保守性
- テストを通すためだけの特殊処理 / 不必要に複雑な実装

問題があれば、影響範囲に応じて工程1〜4まで戻る。

## 工程6: 全テスト実行

変更箇所だけでなく、実行可能な全テストを回す。
- 単体 / 統合 / 回帰
- 静的解析・型チェック・フォーマット・Lint・ビルド
- ドキュメントテスト

**strict のみ追加**:
- カバレッジ計測（新規テストが対象コードを通過しているか）
- セキュリティチェック・依存関係検査
- 実行条件を変える: 単独 / 全体 / 順序変更 / 並列 / キャッシュ無効 / 複数回再実行

**実行範囲（モノレポ等で「全テスト」を確定するための階層）**:
1. 変更パッケージの全テスト
2. 直接の依存・被依存パッケージの回帰テスト
3. リポジトリ標準のCI相当検証（静的解析・型・Lint・ビルド等）
4. 外部環境依存・長時間E2E

原則 1〜3 は実行する。4 は実行可否を評価し、勝手に実行も省略もせず理由を報告する（認証情報・外部環境・数時間規模のものを含む）。実行できなかった項目は「実施済み」にせず理由を明記する。

言語別ツール例:
- Rust: `docker compose -p umbra-rs run --rm rust cargo test --workspace` / `docker compose -p umbra-rs run --rm rust cargo clippy --workspace --all-targets` / `docker compose -p umbra-rs run --rm rust cargo fmt --all --check` /（strict: `docker compose -p umbra-rs run --rm rust cargo llvm-cov` / `docker compose -p umbra-rs run --rm rust cargo mutants`）
- Node/TS: `vitest`(or `jest`) / `tsc --noEmit` / `eslint` / `prettier --check` /（strict: `--coverage`）
- Python: `pytest` / `mypy` / `ruff` /（strict: `pytest --cov`）
- PHP: `phpunit` / `phpstan` / `php-cs-fixer`

## 工程7: テスト結果レビュー

**standard**: 新規テストが実際に対象コードを通過したことを、**実行による客観的証跡**で確認する（自己観点でも証跡は残す）。静的に眺めただけ・レビューのみは不可。次のいずれか:
- 対象テストのみ実行し、ログ・トレースで経路を確認
- 軽量な行カバレッジを取得
- 一時的な観測用ログを入れて確認後に削除

**strict**: テスト作成者・実装者とは**別のサブエージェント**が（standard は別観点での自己レビューでも可）、以下を確認する。
- 偶然通っていないか / 誤った理由で成功していないか
- 新規テストが実際に対象コードを通過しているか（カバレッジで裏取り）
- 重要な分岐が実行されているか
- モック・スタブで問題が隠れていないか
- アサーションまで到達しているか
- スキップ・無効化・条件付き除外されたテストがないか
- 未検証の仕様・コードパスが残っていないか
- 追加すべき回帰テストがないか
- 実行順序・環境に依存していないか / 実環境との差異リスク

### ミューテーション検証（strict・必須ガード）

「重要なテスト」は最低限、新規仕様の中心となる正常系・主要な拒否/異常系・セキュリティ境界・後方互換を保証する回帰ケースを対象とする（バグ修正では元の不具合を再現するテスト）。**1本だけ選んで完了扱いにしない**。

改変は「コンパイルが通り、かつ振る舞いが変わる」ものにする: 条件式の反転 / 境界演算子の変更 / エラー返却の削除 / 状態更新の省略 / 権限確認の無効化 / 戻り値の固定化 など。**適当に壊してコンパイルエラーになっただけではテスト有効性の確認にならない**。

改変方針は次の優先順位に従う:

**専用ツール（`cargo mutants` 等、自前で隔離する） ＞ 本ガード付きの作業ツリー直接改変 ＞ `git worktree`/一時コピーによる隔離**

> 注: `git worktree`・コピーは未追跡・ignore対象ファイル（`.env`・ローカル設定・`target/` 等）を引き継がず cold build になる。数本のスポット確認なら、ガード付き直接改変の方が軽く確実なことが多い。**「一時ブランチ」単体は作業ツリーを共有するため隔離にならない**ので使わない。順序はプロジェクト依存で調整してよい。

作業ツリーを完全にクリーンにする必要はない（red→green を未コミット差分で保持する運用と衝突するため）。代わりに、**検証前の状態を機械的に復元・照合できること**を必須とする。

開始前（必須）:
- `git status --short` / `git diff` / `git diff --cached` で既存差分（未追跡・作業ツリー・ステージ済み）を把握する
- 検証前の差分をパッチ等で退避し、復元可能な状態を確保する（コミット未許可でも退避はする）
- ミューテーション対象外の未追跡・未コミット変更を、**自動で stash・commit・破棄しない**。必要なら人間に確認する

終了処理を実行できる場合（成功・テスト失敗・ツールエラー）は、必ず:
- 一時改変を復元する
- `git status --short` / `git diff` / `git diff --cached` を再確認し、検証前に存在しなかった差分が残っていないことを照合する
- 復元を確認できない場合は、テスト成功として扱わず、作業を停止して人間に報告する

強制終了・再起動・コンテキスト消失・ユーザー中断などで終了処理を実行できなかった可能性がある場合は、**次回作業の冒頭で差分状態を確認し、復元・照合が完了するまで新たな変更を開始しない**。（任意: 改変前に退避パッチと「検証中」一時メモをGit管理外へ置くと、次回の検知が確実になる）

> 限界: 上記でも既存の未追跡ファイルの**内容**変化までは照合しきれない（`status` は存在の列挙のみ）。ツールが未追跡ファイルを書き換えうる場合のみ、退避対象に含める。それ以外は過剰硬化を避ける。

**ツール制約時の扱い**: 対応ツールがない言語・インストール不可・実行時間が非現実的・マクロや生成コードでツールが正常に扱えない等でstrict必須検証を実行できない場合は、代替検証（手動の局所改変による失敗確認など）を行い、未実施理由と残存リスクを報告する。代替でも同等の確認ができない場合は **strict完了とせず**、人間に「リスク受容して完了」か「保留」かの判断を求める。

`cargo mutants` 等を実行した場合も、全mutantが殺せるとは限らない。**生存したmutantは最終報告に列挙し、許容可否を明示する**。

## 工程を戻す条件

該当箇所だけを場当たり的に直さず、必要な工程まで戻る:
- 仕様漏れ → 工程0/1。**確定仕様を変えないと解決できない指摘は、メインエージェントが独断で再解釈しない**。公開挙動・互換性・安全性に影響するなら人間へ確認する。
- テストの誤り・不足 → 工程2/3
- 実装上の問題 → 工程4
- 回帰リスク → 工程1と4の両方を再確認
- テスト結果が不安定 → 原因特定後、テストor実装を修正して工程6を再実行

差し戻し回数が上限に達したら（基本ルールのループ上限を参照）、人間にエスカレーションする。

## 完了条件（モードごと）

**全モード共通**
- 仕様を満たす
- モードに応じて要求された検証が成功している
- 未検証箇所・未実施項目・既知のリスクが明示されている

**standard / strict**
- テストをサブエージェントが設計・作成した
- テストレビューおよび実装レビューで重大な問題がない
- 実行可能な全テストが成功（高コスト等で実行しなかったものは理由を明記）

**strict**
- カバレッジおよびミューテーション検証で、新規テストが正しい理由で有効だと確認できた
- strict必須検証を実行できなかった場合は、人間の判断（リスク受容して完了／保留）を得ている

## 最終報告

- 実装内容 / 採用したモードと理由（standard ならなぜ strict でないか。strict を降格した場合は承認した人間）
- 各サブエージェントの役割
- 追加・変更したテスト
- 実装前のテスト結果（red確認）
- テストレビュー・実装レビューで発見した問題と修正
- 実行したコマンドと結果（全テスト・静的解析・ビルド／strictはカバレッジ等）
- 仕様確定時に置いた仮定
- スキップ・未実施の項目と理由
- （strict）生存したmutantと許容可否 / strict必須検証が実行不能だった場合の人間判断
- 未検証箇所 / 残るリスク / 追加推奨の回帰テスト

**「テストが通った」ことだけを根拠に完了と判断しない。**
